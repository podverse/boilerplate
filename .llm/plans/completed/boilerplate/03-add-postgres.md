# Plan 03: Add Postgres

## Scope

Add a Postgres service to docker-compose and init script(s) in `infra/database/`. Document port
and env vars (e.g. DATABASE_URL or DB_HOST, DB_PORT, DB_NAME, user, password). Create **read-only**
and **read-write** database users (same pattern as Podverse) so apps can use least-privilege
connections. Used by the ORM package (plan 12) and auth (plan 15).

## Steps

1. **Init scripts (order matters)**
   - Mount **two** items into `/docker-entrypoint-initdb.d/` so they run in alphabetical order:
     - **01_create_users.sh** – Creates roles `read` and `read_write` with passwords from env
       (POSTGRES_READ_PASSWORD, POSTGRES_READ_WRITE_PASSWORD). Grants CONNECT and USAGE on
       schema public; GRANT SELECT on all tables/sequences to `read`; GRANT SELECT, INSERT,
       UPDATE, DELETE on tables and SELECT, USAGE, UPDATE on sequences to `read_write`.
       Uses ALTER DEFAULT PRIVILEGES so tables created later by init_database.sql get the
       same permissions. Script must be executable and use `psql -v ON_ERROR_STOP=1` with
       the postgres user. Use a DO block with EXECUTE format(..., password) to avoid
       password in logs. Require env vars with `: "${POSTGRES_READ_PASSWORD:?}"` etc.
    - **init_database.sql** – Schema only (generated by `scripts/database/combine-migrations.sh`
      from numbered files in `infra/database/migrations/`). Plan 12 defines the migration files.
   - The Postgres image already creates the database from POSTGRES_DB; the bootstrap user
     remains POSTGRES_USER/POSTGRES_PASSWORD for admin use only.

2. **Docker service**
   - Add a service named **postgres** (use this exact name so DB_HOST=postgres in env works)
     to the compose file used in plan 02 (e.g. `infra/docker/local/docker-compose.yml` or a
     dedicated db compose).
   - Image: postgres:18.1 (or current LTS).
   - Environment: POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB, **POSTGRES_READ_PASSWORD**,
     **POSTGRES_READ_WRITE_PASSWORD** (required for 01_create_users.sh).
   - Volumes: mount `01_create_users.sh` and `init_database.sql` into
     `/docker-entrypoint-initdb.d/`.
   - Port: expose 5432 to host (e.g. 127.0.0.1:5432:5432).
   - Attach to shared network so api can connect.

3. **Config and env**
   - Add `infra/config/env-templates/db.env.example` with:
     - Postgres container: POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB,
       POSTGRES_READ_PASSWORD, POSTGRES_READ_WRITE_PASSWORD.
     - API/ORM (for app connection): DB_HOST, DB_PORT, DB_NAME; **DB_READ_USERNAME** (e.g. `read`),
       **DB_READ_PASSWORD**; **DB_READ_WRITE_USERNAME** (e.g. `read_write`), **DB_READ_WRITE_PASSWORD**.
     - Optional: DATABASE_URL for drivers that support it (use read-write credentials for
       default connection).
   - Apps use the read-write user for normal API use; the read-only user is available for
     read-heavy or reporting paths if implemented later. Ensure api (and orm) read these
     (e.g. from infra/config/local/db.env or api.env).

4. **Documentation**
   - Update README or infra docs: how to start Postgres, required env vars, that the api/orm
     connect via read and read_write users (not the bootstrap POSTGRES_USER), and that init
     runs 01_create_users.sh then init_database.sql on first start.

## Key files

- `infra/database/combined/01_create_users.sh` (create read + read_write, grant privileges)
- `infra/database/combined/init_database.sql` (generated; source is migrations/ + combine script)
- Compose file that defines the postgres service
- `infra/config/env-templates/db.env.example` (include read/read_write vars)
- README or infra docs

## Verification

- `docker compose up postgres` (or equivalent) starts Postgres; 01_create_users.sh runs first,
  then init_database.sql (check logs). Roles `read` and `read_write` exist; tables have correct
  grants.
- Connect with psql as `read` (SELECT only) and as `read_write` (SELECT/INSERT/UPDATE/DELETE);
  confirm database and users exist.
